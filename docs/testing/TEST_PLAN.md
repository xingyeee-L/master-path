# 测试计划（Master Path）

## 1. 目标
- 验证核心功能满足需求规格：任务铸造、卡组交互、放空计时与自动结算、经验等级、成就系统、双语切换、数据持久化。
- 建立可重复执行的自动化验收：单元测试、组件集成测试、端到端测试。
- 建立质量门禁：类型检查、Lint、构建、依赖安全审计。

## 2. 范围
### 2.1 功能范围
- 任务创建：标题输入、时长选择、卡片生成。
- 卡组展示：3D 圆环、滚轮/拖动旋转、完成按钮（非放空卡）。
- 放空任务：15 分钟倒计时、完成后自动完成并结算 100 XP、当日不重复出现。
- 经验系统：XP 飞行动效、延迟入账、等级进度与头衔（中英）。
- 成就系统：解锁条件、Toast 提示、徽章页集成展示。
- 多语言：右上角切换、中英文案一致性。
- 持久化：IndexedDB（idb-keyval）存取与跨重启一致性。

### 2.2 非功能范围
- 性能：启动/首屏渲染、动画流畅度（主观 + 轻量自动化门槛）、内存泄漏与长时间运行稳定性。
- 安全：依赖漏洞审计、敏感信息泄漏检查（日志/存储/配置）。
- 兼容性：桌面容器（Tauri）与 Web（Chromium/WebKit）基础兼容。

## 3. 测试策略
### 3.1 分层策略
- 单元测试：Store 规则、XP/成就解锁、放空结算与日期逻辑。
- 集成测试：关键组件（徽章页成就展示、等级头衔双语）。
- 端到端：用户路径（打开应用、语言切换、创建任务、完成任务触发 XP 动效、进入放空倒计时）。

### 3.2 覆盖目标
- 关键业务逻辑行覆盖率：≥ 70%（Store 与 i18n 优先）。
- 自动化通过率：100%（Typecheck/Lint/Test/Build/Audit 作为门禁）。

## 4. 环境与工具
- Node.js：与项目一致（本机/CI）。
- 测试框架：Vitest + Testing Library。
- E2E：Playwright（Chromium / WebKit）。
- 静态分析：TypeScript、ESLint。
- 安全：npm audit。

## 5. 进入/退出准则
### 5.1 进入准则
- 功能合并至主分支前，必须通过：typecheck、lint、unit/integration tests。

### 5.2 退出准则
- `npm run typecheck` 通过
- `npm run lint` 通过
- `npm test` 通过
- `npm run build` 通过
- `npm run audit` 0 vulnerabilities
- E2E 用例通过（需 Playwright 浏览器安装完成）

## 6. 风险与应对
- Canvas/动画导致测试不稳定：测试中对动画/外部库进行 mock，E2E 覆盖实际运行。
- IndexedDB 在 jsdom 不可用：单元测试 mock `idb-keyval`，E2E 覆盖真实浏览器。
- 长时间计时（15 分钟）难以自动化：E2E 覆盖进入/展示，单元测试覆盖完成回调逻辑与 XP 结算。
